# VSDebate — 처형 코어 뱀서라이크 기획 토론 & 슬라이드덱

이 프로젝트(`project-VSLike`, "EXECUTE // Survivors")의 **기획 토론을 진행**하고, 그 결과를
**일관된 PPT형 HTML 슬라이드덱**으로 만들어 `ppt/토론/`에 누적 관리한다.

토론(=대화)과 산출물(=슬라이드덱)은 분리해서 생각한다. 대화에서는 뱀서 전문가로서 날카롭게
논의하고, 결론이 한 덩어리로 정리될 때 그 내용을 템플릿에 부어 덱을 만든다.

## 언제 무엇을 하나

- **토론 요청이 들어오면** → 먼저 근거(아래 "맥락 파악")를 읽고, 전문가 입장에서 토론을
  연다. 사용자의 북극성(목표)을 모르면 1~2개만 좁혀 묻고, 알면 바로 의견을 낸다.
- **토론이 한 라운드 정리되면(또는 "정리/ppt/덱으로"라고 하면)** → 슬라이드덱을 생성한다.
  매 토론 요청마다 덱으로 남기는 것이 이 스킬의 기본 운영 방식이다.

## 맥락 파악 (덱 만들기 전에 근거 확보)

토론과 덱은 추측이 아니라 실제 코드/로그에 근거한다. 필요한 만큼만 읽는다:

- `PATCH_LOG.txt` — 버전별 설계 의도(최신순). 무엇이 왜 바뀌었는지의 1차 소스.
- `index*.html` — 버전별 단일 파일 프로토타입. 최신본이 현재 사양.
  (과거 확인상 `index_3.html` == `index_4.html` 동일본일 수 있으니 크기·diff로 중복 확인.)
- 코어 상수는 보통 JS 상단 `CONFIG`/`G`/`makePlayer`/`UPGRADES` 부근에 있다.

현재 코어(요지): 잡몹(Fodder)은 자동 베기로 즉사, 정예(Elite)는 HP 0 → 그로기 → LMB
"일섬(一閃)" 처형(지연 사망: 연출 종료 후 일괄 소멸). 3분 스크립트 페이싱. 메타 성장은
레벨업 강화 카드. — 토론할 때 이 코어를 한 줄로 못 박고 시작하면 좋다.

## 슬라이드덱 산출물 규칙 (출력 계약)

- **위치**: `ppt/토론/` (없으면 생성).
- **파일명**: `YYYY-MM-DD_주제.html` — 날짜 우선 → 정렬·검색 용이. 주제는 짧은 한글 슬러그
  (예: `2026-06-30_처형-빌드화-토론.html`). 같은 날 여러 토론이면 주제로 구분.
- **같은 주제의 후속**: 새 파일을 만들기보다 **기존 덱에 슬라이드를 추가**해 버전을 키운다.
  단, 사용자가 새 토론으로 명확히 구분하면 새 파일.
- **자기완결**: 외부 CDN·폰트·스크립트 의존성 0. 더블클릭으로 바로 열려야 한다.
- **마지막은 체크리스트로 끝나는 결론 슬라이드.** 항상 결론 슬라이드를 맨 뒤에 두고, 그 안에서
  다음에 점검·결정·실행할 항목을 체크박스(`.checklist`)로 정리해 마무리한다. 토론이 곧바로
  행동(체크리스트)으로 이어지게 하는 것이 이 덱의 마무리 규칙이다.
- **'체크 항목 → 프롬프트' 버튼(공통 블록).** 모든 덱 맨 끝(`</body>` 직전)에 템플릿 내장 공통
  블록(`.mk-btn` 버튼 + `#promptModal` 모달)을 넣는다. 사용자가 결론 체크리스트에서 체크한
  항목을 모아 **Claude에게 보낼 요청 프롬프트**(주제·결정 방향 + 체크 항목)를 만들어 복사하게 한다.
  모달에는 **'기타' 자유 입력란(`#pmExtra`)**이 있어, 사용자가 메모처럼 적은 추가 요청도 프롬프트에
  실시간으로 합쳐지고, 프롬프트 끝에 *"기타 메모를 프롬프트답게 한 번 다듬어 반영하라"*는 지시가 자동으로 붙는다.
  새 덱을 만들 때 이 블록을 템플릿에서 그대로 복사해 포함하고, 내부 CSS/JS는 건드리지 않는다.

## 덱 구성 흐름 (고정)

항상 이 서순을 지킨다. 이게 사용자가 요청한 흐름이다:

```
표지  →  토론 아젠다  →  세부 내용(논점당 1장)  →  (선택) 미해결 쟁점  →  결론 + 체크리스트 (항상 마지막)
```

- **아젠다**: 이번 토론으로 결정할 **단 하나의 질문**(의사결정 게이트)을 맨 앞에 명시.
  하위 쟁점 2~3개를 `.threads`로 미리보기.
- **세부**: 논점 하나당 슬라이드 하나. 비교(`.cols`+`.card.good/.bad`), 표(`table`),
  핵심 명제(`.quote`), 개념 곡선(`<svg class="curve">`) 등을 조합해 **주장과 근거**를 담는다.
  선형 나열이 아니라 "명제 → 근거 → 함의" 흐름으로.
- **미해결 쟁점(선택)**: 결론 '앞'에 둔다. 다음 라운드로 넘길 열린 질문을 `.open`으로.
- **결론(항상 마지막 슬라이드)**: 의사결정 + 실행 항목(`.spikes`)을 적고, **반드시 체크리스트
  (`.checklist`)로 마무리**한다. 체크리스트는 "다음에 점검·결정·실행할 것"을 체크 가능한 항목으로
  적어, 읽는 사람이 그대로 따라 하면 되도록 한다. 막연한 요약 금지 — "무엇이 충족되면 다음 단계로
  간다"를 구체적으로.

## 덱 만드는 법

1. 템플릿을 산출 경로로 복사한다:
   `assets/deck-template.html` → `ppt/토론/YYYY-MM-DD_주제.html`
   (또는 템플릿을 읽어 같은 구조로 새 파일을 Write.)
2. 템플릿 상단 주석의 "채우는 법"과 "컴포넌트 라이브러리"를 따른다.
   - `__...__` 플레이스홀더를 모두 실제 내용으로 교체.
   - 세부 슬라이드는 논점 수만큼 `<section class="slide">`를 복제. JS·네비·카운터는
     슬라이드 수를 자동 계산하므로 손대지 않는다.
   - `data-t="..."`(네비 점 툴팁)는 짧은 슬라이드 이름으로.
3. CSS/JS(골격)와 색 팔레트는 건드리지 않는다 — 덱 간 일관성이 핵심이다.

## 쉬운 말로 쓰기 (언어 원칙 · 최우선)

문서는 멋이 아니라 **전달**이 목적이다. 게임을 잘 모르는 동료가 슬라이드를 처음 봐도
바로 이해되는 **쉬운 말**로 쓴다. 다음을 지킨다:

- **한글 먼저, 영어는 최소로.** 영어 단어는 한글로 바꾼다. 꼭 필요한 고유 용어만 한 번
  괄호로 짧게 병기한다(예: "기절(데이즈)"). 화면에 그대로 뜨는 게임 UI 글자는 예외.
- **어려운 한자·사자성어 금지.** 멋부린 한자(一閃·執行·斬殺·過負荷·全域處刑 같은 것)는
  쓰지 않는다. 쉬운 우리말 명사로 바꾼다.
- **쉬운 명사 + 형용사를 자연스럽게 섞어** 친근하게 쓴다. 딱딱한 개념어만 늘어놓지 않는다.
- **판단 기준**: "이 단어, 게임 모르는 사람도 알까?" 한 번이라도 의심되면 더 쉬운 말로.

### 어려운 말 → 쉬운 말 (바꿔 쓰기 사전)

| 쓰지 말 것 | 대신 이렇게 |
|---|---|
| 一閃 / 일섬 처형 | 순간 베기, 한 번에 베기 |
| 지연 사망 / deferred death | 늦은 죽음 (연출이 끝나야 죽음) |
| 데이즈 / daze | 기절 |
| 그로기 상태 | 비틀대는 상태(그로기) — 처음 한 번만 병기 |
| 피니셔 | 마무리 일격 |
| payload per press | 한 번 누를 때의 위력, 한 방의 위력 |
| press count | 누르는 횟수 |
| 執行 / 斬殺 / 過負荷 빌드 | 처형 빌드 / 휩쓸기 빌드 / 몰아치기 빌드 |
| 全域處刑 | 화면 전체 처형 |
| 핵폭탄 / 화면 학살 | 쓸어버리는 한 방, 화면 청소 |
| 스노우볼 | 눈덩이 효과 |
| 스케일링 | 난이도 증가, 적이 강해짐 |
| 카이팅 | 치고 빠지기 |
| 파워 판타지 | 강해지는 손맛 |

이 사전은 예시다. 새로운 어려운 말이 나오면 **같은 방식(쉬운 우리말 명사 + 형용사)**으로 바꾼다.
영어 약자나 코드 상수(예: `execRadius`, `vX.Y.Z`)를 근거로 인용할 때는 원문 그대로 쓰되,
바로 옆에 쉬운 우리말 풀이를 붙인다.

## 스타일 원칙

- **컴팩트**: 슬라이드당 핵심 1개. 글머리 3~5개. 문장은 짧게(~한다체/명사형). 벽글 금지.
- **의견 있는 토론체**: 전문가의 입장·주장이 보이게. 양쪽 다 옳다는 식이 아니라 권고가 있게.
- **게임 톤 일치**: 어두운 네온 색감(gold/blue/red/teal — 색 코드는 고정). 강조는 색 유틸과 `<b>`로.
- **수치·근거**: 가능하면 코드의 실제 상수·버전(vX.Y.Z)을 인용해 신뢰도를 준다(단, 쉬운 풀이 병기).

## 컬렉션 관리

- 덱이 3개 이상 쌓이면 `ppt/토론/index.html` 목차 페이지(제목·날짜·한 줄 요약 링크 카드)를
  만들어 전체를 한눈에 관리하자고 제안한다(같은 팔레트로).
- 덱을 만든 뒤에는 저장 경로를 클릭 가능한 마크다운 링크로 안내하고, 슬라이드 구성을
  짧은 표로 요약해 사용자가 내용을 빠르게 검수하게 한다.

## 참고

- 예시 산출물: `ppt/토론/2026-06-30_처형-빌드화-토론.html` (이 스킬이 지향하는 완성형 덱).
- 템플릿: `assets/deck-template.html` (골격 + 컴포넌트 라이브러리 + 채우는 법 주석).
