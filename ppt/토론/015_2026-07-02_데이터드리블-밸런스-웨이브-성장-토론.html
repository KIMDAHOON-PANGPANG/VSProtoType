감이 아니라 규칙으로 굴릴 수 있나?

"성장·등장 숫자를 전부 공식(커브)과 게이트(잠금)로 바꿔 JSON 한 곳에서 드리블할까?"

이미 라운드 방식(웨이브+상점)은 정했다(덱 09·14). 이번엔 그 안을 무엇을 어떤 모양의 데이터로 뺄지를 못 박는다. 지금은 spawner.phase1/2/3 시간표가 코드에 반쯤 박혀 있어 손대기 번거롭다.

02 · 세부 — 몬스터 수량 밸런스

①웨이브 = 표 한 줄, 밀도는 예산 ÷ 길이

핵심: interval(스폰 간격)을 감으로 만지지 말고, "이 웨이브에 총 몇 마리(예산)"를 먼저 정한 뒤 간격을 역산한다. 브로테이토처럼 웨이브가 표의 한 행이 된다.

// gameBalance.json — spawner.phase* 를 waves[] 배열로 승격 "waves": [ { id:1, duration:45, budget:55, mix:{fodder:1.0}, elite:0 }, { id:2, duration:50, budget:80, mix:{fodder:.85, runner:.15}, elite:1 }, { id:5, duration:60, budget:120, mix:{fodder:.7, runner:.3}, boss:"knight_captain" } ]

= 초당 1.2마리 (역산된 간격 0.82s)

현재값(참고): phase1 interval 0.62·cap 120, phase3 interval 0.26·cap 340·elite 최대 3 → 이걸 waves 표의 1행씩으로 옮기면 코드 수정 없이 숫자만 드리블.

03 · 세부 — 웨이브별 HP 스케일링

②HP는 곱셈 커브 한 줄로 자란다

선형(더하기)

웨이브마다 같은 양 +. 후반에 무기 화력(곱셈)에 뒤처져 너무 물러진다.

hp = base × (1 + k·(w−1))
추천 →

지수(곱하기)

무기도 레벨로 곱셈 성장 → 적 HP도 곱셈이라야 추격이 된다. 단 mult 작게(1.10~1.15).

hp = base × mult^(w−1)

데이터: 적별 hpMult 하나 + 보스 웨이브만 bossMult로 계단 점프. 예) fodder base 10·mult 1.12 → 웨이브10 ≈ 27, 웨이브15 ≈ 48. elite base 50 → 웨이브10 ≈ 138. 커브는 단순하게, 놀람은 보스에서.

04 · 세부 — 회피·HP·이속·아머, 왜?

③스탯은 '다른 커브'일 때만 넣는다

스탯 자체가 목적이 아니다 — 스탯은 장신구 빌드의 재료다. 없으면 장신구가 전부 '공격력 +'만 되어 빌드가 단조로워진다. 판정 기준: "이미 있는 커브와 겹치면 뺀다."

스탯여는 축우리에 필요?판정 · 이유
HP실수 버퍼(맞아도 버팀)유지코어 생존치. 최대HP·전체회복 카드 이미 존재.
회피(대시)능동 회피 · 저스트유지우리 정체성(2층 방어). 쿨·무적시간 카드 축.
이동속도치고 빠지기(카이팅)소량떼거리 도는 재미. 단 과하면 위협 무력화 → 조금만.
아머받는 피해 감소조건부HP와 커브가 겹침. '받는 피해 −%'로 달라질 때만 도입, 아니면 보류.
아머를 넣을 거면 HP와 다른 모양(비율 감소 vs 고정 버퍼)이어야 존재 이유가 생긴다.
05 · 세부 — 무기·장신구·처형 빌드 성장

④패시브는 매 레벨, 액티브는 레벨 게이트

패시브 (수치)

공격력·틱 데미지·개수 — 매 레벨 선형으로 부드럽게. 이미 weapons.json이 이 방식.

실효 = base + (Lv−1) × perX
+

액티브 (새 효과)

새 무기 습득·진화·특수효과 — 드문 '와우 순간'. 정해진 레벨에서만 연다.

Lv 5 / 10 / 15 게이트
Lv1무기 습득
→
Lv2~4패시브만 성장
→
Lv5액티브 1 개방
→
Lv6~9패시브
→
Lv10진화 / 액티브 2

현행 perX(참고): homing +4/레벨, star +5, bolt +6, 처형 파문 shock +16. → 패시브는 이대로 두고, 각 무기 JSON에 evolveAt:5 같은 액티브 게이트 필드만 추가하면 두 축이 분리된다.

06 · 세부 — 장신구는 언제부터?

⑤등장도 게이트 숫자 하나로

VS도 초반엔 무기만, 장신구는 나중에 합류한다. 이유는 초반에 '무기 정체성'부터 세우게 하려는 것. 이걸 감이 아니라 아이템마다 unlockWave 숫자로 박는다.

웨이브 1~2무기만 (정체성)
→
웨이브 3+장신구 합류
→
웨이브 5+전직·특수 각인
// 각 아이템에 게이트 필드 → 카드 뽑기 때 필터 "talisman_haste": { stat:"moveSpeed", unlockWave:3 } // 뽑기: pool.filter(it => wave >= (it.unlockWave || 1))
밸런스 드리블 가능? 그렇다. 숫자 하나(unlockWave)만 바꾸면 등장 타이밍이 통째로 움직인다 — 코드 수정 0.

accessories.json(공용 스탯 15종)은 이미 대기(KIP) 중 → 각 항목에 unlockWave만 붙이면 끝.

07 · 세부 — 매직서바이버식 상자→상점

⑥초를 찍지 말고 웨이브 경계에 붙인다

"몇 초가 적당?"의 답은 감으로 초를 찍는 게 아니라, 이미 있는 웨이브 경계(긴장의 골짜기)에 붙이는 것. 그러면 웨이브 길이(45~75초)가 자동으로 간격이 된다.

◆기본: shopEvery:1 (매 웨이브 클리어 직후) ◆중간 상자 원하면: midChest:{atFraction:0.6} (웨이브 60% 지점) ◆재화 = 골드 젬(덱 09) 승격
08 · 토론 계속 — 미해결 쟁점

다음 갈림길

쟁점 A

아머, 넣을까 뺄까

HP와 커브가 겹친다. '받는 피해 −%'로 확실히 다르면 O, 애매하면 초반엔 X(보류).

쟁점 B

HP 커브 상한

지수는 좋지만 엔드리스(20분 연속) 모드에선 폭주 위험. 웨이브 N 이후 상한(캡)을 둘까?

쟁점 C

웨이브 중간 상자

경계 상점만으로 충분한가, 아니면 매직서바이버처럼 웨이브 도중 상자도 실험할까?

⮕ A·B·C를 정하면 waves 테이블의 컬럼(아머·hpCap·midChest)이 확정된다.
